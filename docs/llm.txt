# Bifrost CLI & MCP Command Reference (for LLMs)

This file documents the entity-mutation surface Claude and other LLMs should use when the user asks for anything that would previously have been done by editing `.bifrost/*.yaml` by hand.

**Core rule:** `.bifrost/` is export-only. Never read it for discovery and never mutate entities by editing it. For discovery (before mutating), use `bifrost <entity> list` and `bifrost <entity> get <ref>` — the manifest is an export artifact, not the source of truth queries should hit.

All CLI commands accept `--json` for machine output. All commands resolve UUID / name / portable refs automatically; ambiguous names fail loudly with the candidate list.

**Flags are generated from Pydantic DTOs.** The authoritative per-command flag reference is always `bifrost <entity> <verb> --help` — this is generated live from `XxxCreate` / `XxxUpdate` at runtime, so it cannot drift. This file documents commands and non-obvious semantics only; it does not duplicate the flag surface.

---

## Organizations

Tenant records that scope users, configs, and most other entities.

Commands: `orgs list / get / create / update / delete`.

Example: `bifrost orgs create --name "Acme"`.

Run `bifrost orgs <verb> --help` for the complete flag list.

## Roles

Named permission buckets assigned to workflows / forms / agents / apps for role-based access control.

Commands: `roles list / get / create / update / delete`.

Example: `bifrost roles create --name "Sales Manager" --permissions @perms.yaml`.

Non-obvious semantics:
- `permissions` is a dict (not a list). Pass JSON inline or `@path/to/file.yaml`.

Run `bifrost roles <verb> --help` for the complete flag list.

## Workflows

Registered Python functions available for execution by forms / agents / events / HTTP endpoints.

Commands: `workflows list / get / register / update / delete / grant-role / revoke-role`.

Example: `bifrost workflows register --path workflows/foo.py --function-name foo`.

Non-obvious semantics:
- `<workflow-ref>` accepts UUID, name, or `path::func` (e.g. `workflows/sales.py::close_deal`).
- `get` is served by list-and-filter (the API has no per-id GET endpoint).
- `register` is how workflows come into existence — workflows are never "created" in the CLI sense because they're defined by a `@workflow`-decorated Python function. Write the `.py` file first, then `register` it (or sync it via `bifrost watch`).
- `delete --force-deactivation` bypasses the 409-on-history guard.

### Renaming or moving a workflow

If you rename a `@workflow` function or move its file without editing `.bifrost/workflows.yaml`, the indexer marks the row `is_orphaned=True` and skips the "new" location. Explicit re-registration mints a **new UUID**, breaking every form / agent / app reference that pointed at the old one. Use `replace` instead to preserve the UUID:

```bash
# 1. Find orphaned UUIDs (active workflows are not shown here)
bifrost workflows list-orphaned --json

# 2. Repoint the orphaned UUID at the new file location
#    The file must already exist and contain the named decorated function.
bifrost workflows replace <uuid> --path workflows/new_path.py --function-name new_func

# Type changes (@workflow → @data_provider etc.) are blocked by default.
# Pass --allow-type-change only if you intentionally changed the decorator.
bifrost workflows replace <uuid> --path workflows/new_path.py --function-name new_func --allow-type-change
```

`replace` validates that the target file contains a `@workflow` / `@tool` / `@data_provider` decorated function with the given name before committing — it will error if the file is missing or the function lacks a decorator.

Run `bifrost workflows <verb> --help` for the complete flag list.

## Forms

User-facing parameter collection UIs that launch a workflow on submit.

Commands: `forms list / get / create / update / delete`.

Example: `bifrost forms create --name onboard --workflow path::func --form-schema @schema.yaml`.

Non-obvious semantics:
- `--form-schema` accepts YAML/JSON inline or `@path/to/schema.yaml`. The schema is a `{fields: [...]}` object; run `bifrost forms --help` or look at an exported `.bifrost/forms.yaml` entry for the shape.
- `--workflow` / `--launch-workflow` accept portable refs (UUID, name, or `path::func`).

Run `bifrost forms <verb> --help` for the complete flag list.

## Agents

LLM-backed conversational entities with configurable tools, delegations, knowledge sources, and system prompts.

Commands: `agents list / get / create / update / delete`.

Example: `bifrost agents create --name support --system-prompt @prompt.md --tool-ids wf1,wf2`.

Non-obvious semantics:
- `update` uses PUT (full replacement of `tool_ids` / `delegated_agent_ids` / `knowledge_sources` lists — not merge).
- `--system-prompt @file.md` loads a multi-line prompt from disk.
- `--tool-ids`, `--delegated-agent-ids`, `--role-ids` accept comma-separated refs; each entry is resolved to a UUID (workflows for tools, agents for delegations, roles by name).
- `--clear-roles` wipes all role assignments.

Run `bifrost agents <verb> --help` for the complete flag list.

## Apps

App Builder applications (TSX/TypeScript source + npm dependencies).

Commands: `apps list / get / create / update / set-deps / replace / delete`.

Example: `bifrost apps create --name dashboard --slug dashboard --deps @package.json`.

Non-obvious semantics:
- `create --deps` is a two-call orchestration (POST app, then PUT dependencies). If the deps call fails the app remains created; the user sees both the app payload and the deps error.
- `get` accepts slug, UUID, or name. Slug uses the per-record endpoint directly; UUID/name resolves then locates in the list.
- `update` is patch-without-draft — metadata changes apply to the live app, no staging step.
- `replace <ref> --repo-path <new>` repoints an app's source directory. Validates uniqueness (no other app already owns `<new>`), nesting (no other app's `repo_path` sits above or below `<new>`), and that source files exist under `<new>`. `--force` bypasses all three. Metadata-only: no S3 file moves — caller reconciles files via `bifrost sync`.

Run `bifrost apps <verb> --help` for the complete flag list.

## Integrations

External service definitions (HaloPSA, Microsoft Graph, etc.) with config schemas and per-org mappings carrying OAuth credentials / API keys.

Commands: `integrations list / get / create / update / add-mapping / update-mapping`.

Example: `bifrost integrations create --name halopsa --config-schema @schema.yaml`.

Non-obvious semantics:
- `update --config-schema` refuses to remove keys from the schema unless `--force-remove-keys` is set (Config rows cascade-delete when schema rows are removed).
- `update-mapping` never touches `oauth_token_id` unless it's explicitly passed via `--oauth-token-id` — avoids clobbering tokens set by the OAuth flow in the UI.
- `--organization` on mappings resolves by org ref (UUID or name).

Run `bifrost integrations <verb> --help` for the complete flag list.

## Configs

Key-value configuration entries (global or per-org), optionally encrypted (`secret` type).

Commands: `configs list / get / create / update / set / delete`.

Example: `bifrost configs set api_key --value xyz --type secret --organization acme`.

Non-obvious semantics:
- `set <key>` is an upsert wrapper: routes to PUT if `(key, org)` exists, POST otherwise.
- `get` is served by list-and-filter (the API has no per-id GET endpoint).
- `update` with `--value` omitted preserves the existing (encrypted) value.
- `delete` on a secret-type config refuses unless `--confirm` is set.

Run `bifrost configs <verb> --help` for the complete flag list.

## Tables

Document-store tables (JSON documents, optional schema hints).

Commands: `tables list / get / create / update / delete`.

Example: `bifrost tables create --name clients --application app-ref --schema @schema.yaml`.

Non-obvious semantics:
- `update --name` prints a warning to stderr about workflow SDK references (SDK calls like `sdk.tables.get("clients")` reference tables by name). It does not block — the author must grep the workspace before pushing.
- `--application` accepts app slug / UUID / name.

Run `bifrost tables <verb> --help` for the complete flag list.

## Events

Event sources (webhook / schedule / internal) and the subscriptions that dispatch events to workflows or agents.

Commands: `events list-sources / get-source / list-subscriptions / get-subscription / create-source / update-source / subscribe / update-subscription`.

Examples:
- `bifrost events create-source --name nightly --source-type schedule --cron "0 2 * * *" --timezone UTC`
- `bifrost events create-source --name hook --source-type webhook --adapter generic`
- `bifrost events subscribe <source-ref> --workflow <wf-ref> --event-type ticket.created`
- `bifrost events update-subscription <source-ref> <subscription-id> --event-type ticket.updated`

Non-obvious semantics:
- `update-subscription` takes **two** positional args (`<source-ref> <subscription-id>`), matching the REST path `/api/events/sources/{source_id}/subscriptions/{subscription_id}`.
- `--cron` / `--timezone` / `--schedule-enabled` collapse into the DTO's nested `schedule` config.
- `--adapter` / `--webhook-integration` / `--webhook-config` collapse into the nested `webhook` config.
- `update-subscription` rejects changes to `target_type` / `workflow_id` / `agent_id` — delete and recreate instead (delivery history is tied to the target).
- `get-subscription` is served by list-and-filter (the API has no per-id GET endpoint for subscriptions).
- `subscribe` accepts exactly one of `--workflow` or `--agent`; target type is inferred.

Run `bifrost events <verb> --help` for the complete flag list.

## Cross-environment bundles

```
bifrost export --portable <out-dir>      # scrubs org_ids, user_ids, timestamps, OAuth secrets, secret values; rewrites role_ids → role_names
bifrost import <bundle-dir> --org <target-uuid> [--role-mode name|uuid] [--dry-run] [--delete-removed]
```

`export --portable` writes `<out-dir>/.bifrost/*.yaml`, `<out-dir>/workflows/`, `<out-dir>/apps/`, and `<out-dir>/bundle.meta.yaml` (source env, bifrost version, scrub rules applied).

`import` uploads workflow/app code via `POST /api/files/write` first, then POSTs `.bifrost/` content to `/api/files/manifest/import` with the target org UUID. `--role-mode name` resolves roles by name in the target env; missing role names fail loud.

## MCP tools (parity with CLI)

Existing MCP tools for agents / forms / tables / apps / events re-implement router logic and have diverged. See `docs/plans/2026-04-18-mcp-router-reconciliation.md`.

**New parity tools (thin HTTP wrappers — always safe to call):**

- `list_roles`, `get_role`, `create_role`, `update_role`, `delete_role`
- `list_configs`, `get_config`, `create_config`, `update_config`, `delete_config`
- `get_integration`, `create_integration`, `update_integration`, `add_integration_mapping`, `update_integration_mapping`  (`list_integrations` already existed)
- `update_organization`, `delete_organization`  (list / create / get already existed)
- `update_workflow`, `delete_workflow`, `grant_workflow_role`, `revoke_workflow_role`  (list / get / register already existed)

All new tools accept UUID, name, or portable ref for any entity reference. MCP tool parameters are generated from the same `XxxCreate` / `XxxUpdate` DTOs as the CLI flags, so the two surfaces stay field-parallel.

## Escape hatch

For anything without a dedicated command:

```
bifrost api <METHOD> <path> [json-body]
```

Authenticated REST passthrough. Use sparingly — dedicated commands give better ergonomics and ref-resolution errors.

---

## Do NOT

- Edit `.bifrost/*.yaml` by hand to mutate entities. That's export-only.
- Use the (removed) granular agent endpoints `/api/agents/{id}/tools` or `/delegations`. Use `PUT /api/agents/{id}` with full `tool_ids` / `delegated_agent_ids` lists.
- Add direct ORM access to new MCP tools. Thin HTTP wrappers only; enforced by `api/tests/unit/test_mcp_thin_wrapper.py`.
- Duplicate flag lists in this file or in any skill doc. Trust `--help`; it is generated from the DTO and cannot drift.
